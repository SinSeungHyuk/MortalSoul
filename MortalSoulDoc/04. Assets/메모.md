# 몬스터 공격 / 스킬 시스템 설계 논의 (진행 중)

## 배경 / 문제의식

- 처음엔 몬스터는 기본공격 없이 스킬만으로 구현하려 했음
- 그런데 유저가 보기엔 몬스터의 "스킬"도 사실상 기본공격처럼 보임
- 몬스터 공격도 즉시 발동이 아니라 **Spine 애니메이션 구독(키프레임)** 을 통해 진행되어야 함
- 단, 몬스터는 무기를 가진 개념이 아니라서 WSC에 묶기엔 부적절
- 한 몬스터 내에서도 **어떤 공격은 Spine 이벤트 기반, 어떤 공격은 즉시 발동형**이 섞여 있음 (예: 근접 베기는 hit 이벤트 대기, 투사체 소환은 즉시 발동)
- 몬스터는 1~3개의 공격 패턴을 가지며, 이전 레퍼런스처럼 **Rate 기반 랜덤 선택** 구조 예정

## 결정된 방향

### 1. 몬스터 공격 = SSC의 스킬로 처리
- 새 컴포넌트(MAC 등) 만들 필요 없음
- SSC는 그대로 사용, Rate 기반 랜덤 선택만 추가
- 플레이어의 기본공격(WSC)은 기획상 스킬이 아니므로 그대로 유지 (WSC는 플레이어 전용)

### 2. BaseSkill에 Spine 이벤트 기반 타이밍 헬퍼 추가 (접근 방식 A 채택)
- SkillSettingData에 플래그 두고 템플릿 메서드로 가는 방식(B안) 대신, BaseSkill이 **헬퍼 메서드만 제공**하고 각 스킬 서브클래스가 자유롭게 조립
- 이유:
  1. 이미 리플렉션으로 개별 스킬 클래스를 만드는 구조라 "스킬마다 코드" 부담은 이미 받아들인 상태
  2. 보스 패턴·고유 스킬의 복잡도(다단 히트, 투사체+히트 혼합, 조건 분기)를 템플릿으로 가두면 금방 한계
  3. SSC 변경 없음, BaseSkill에 헬퍼만 추가하면 되므로 변경 범위 최소

**BaseSkill 헬퍼 예시**:
```csharp
protected UniTask PlayAnimAndWaitHit(string animKey, CancellationToken ct);
protected UniTask PlayAnimAndWaitComplete(string animKey, CancellationToken ct);
protected UniTask WaitForHit(CancellationToken ct);       // 2단 히트, 복합 스킬용
protected UniTask WaitForComplete(CancellationToken ct);
```

**사용 예**:
```csharp
// 즉시형 스킬 (투사체)
await PlayAnimAndWaitComplete("Attack_Fireball", ct);
SpawnProjectile();

// 히트형 스킬 (근접 베기)
await PlayAnimAndWaitHit("Attack_Slash", ct);
DealDamage();
await WaitForComplete(ct);
```

### 3. 공통 인터페이스 ISpineController 도입
- 플레이어에는 `PlayerSpineController`가 이미 존재, 몬스터도 자기 SpineController 필요
- `BaseSkill`이 owner의 SpineController에 접근해야 하므로 공통 인터페이스로 추상화

```csharp
interface ISpineController {
    event Action OnHitEvent;
    event Action OnComboReadyEvent;
    event Action OnActionCompleted;
    void Play(string animKey, bool loop);
}
```

- `FieldCharacter`가 `ISpineController`를 노출 → `BaseSkill`이 `owner.SpineController`로 접근
- 플레이어 WSC도 이 인터페이스로 이벤트 구독 통일 (지금은 WSC 미구현이라 설계 단계에서 맞추면 됨)

### 4. 애니메이션 재탕으로 스킬화 가능 (A안의 큰 장점)
- **애니메이션 = 단순 리소스 키**. 누가 구동하느냐에 따라 기본공격/스킬로 달라짐
- 같은 `BasicAttack_Swing_1` 애니를 스킬이 재생하면 스킬 데미지/이펙트가 적용됨
- 데미지 계수·판정 범위·이펙트는 각자의 데이터(WeaponSettingData vs SkillSettingData)에서 읽음
- **이점**:
  - 리소스 한계 돌파 — 같은 애니로 등급 다른 스킬 여러 개 파생
  - "N타 기본공격 애니를 그대로 쓰는 피니셔 스킬" 자연스럽게 구현
  - 몬스터는 애니 셋만 맞추면 리소스 1개로 공격 2~3개 파생

### 5. 이벤트 처리: 단일 이벤트 + 내부 상태 가드 (구독 수명 관리 불필요)
- 처음엔 try/finally로 구독 수명 관리하려 했으나 오버엔지니어링
- 중간에 "키프레임에 `attack_hit` / `skill_hit` 두 이벤트 분리" 아이디어 나왔으나 **근본 문제 해결 안 됨**:
  - 애니 클립은 자기가 왜 재생되는지 모름 → 두 이벤트 모두 발사 → WSC가 스킬 재탕 애니에서도 기본공격 데미지 쏘게 됨
  - 애니메이터가 공유 애니마다 두 이벤트 빠짐없이 박아야 하는 작성 부담
- **최종 해법**: **단일 `hit` 이벤트, 양쪽 상시 구독, 각자 내부 상태 가드**

```csharp
// WSC: 이미 콤보 상태 플래그 있으니 가드만 추가
void HandleHit() {
    if (!isComboActive) return;   // 스킬이 애니 돌리는 중이면 무시
    // ... 기본공격 히트 처리
}

// BaseSkill: 대기 중인 히트 TCS가 있냐만 체크
private UniTaskCompletionSource pendingHitTcs;

void HandleHit() {
    pendingHitTcs?.TrySetResult();
    pendingHitTcs = null;
}

protected async UniTask PlayAnimAndWaitHit(string key, CT ct) {
    pendingHitTcs = new UniTaskCompletionSource();
    spine.Play(key, false);
    await pendingHitTcs.Task.AttachExternalCancellation(ct);
}
```

**왜 깔끔한가**:
- `+=` / `-=` / try/finally 전부 없음. 구독은 초기화 시점 한 번
- WSC는 이미 가진 콤보 상태를 가드로 쓰니 추가 비용 0
- BaseSkill은 `pendingHitTcs == null`이면 자연 무시
- 애니메이터는 `hit` 이벤트 하나만 찍으면 됨
- 게임 기획(공격과 스킬 동시 불가)이 `isComboActive == true && pendingHitTcs != null` 동시 참을 원천적으로 막아줌

## 남은 논의 사항 (다음 대화에서 이어서)

### Rate 기반 몬스터 스킬 선택 로직 위치
- **옵션 1**: SSC 안에 통합 — SSC에 `UseRandomSkill()` 메서드, Rate 테이블도 SSC가 관리. 플레이어는 키 직접 지정, 몬스터는 랜덤 선택
- **옵션 2**: 몬스터 AI 레벨 — 별도 `MonsterAI`/`MonsterBrain`이 Rate 로직 돌려 `SSC.UseSkill(key)` 호출. SSC는 선택 로직 모름 (관심사 분리상 깔끔)
- **옵션 3**: MonsterCharacter 자체가 보유 — AI 없이 직접 관리
- 현재 프로젝트에 `MonsterAI` 류 개념이 있는지 확인 필요

### 기타 검토 필요
- MonsterSpineController 신규 작성 필요
- 몬스터 스킬의 쿨타임 + Rate 복합 선택 로직
- 2단 히트·복합 타이밍 스킬의 실제 사용 사례 확인
