# 몬스터 공격 / 스킬 시스템 설계 논의 (진행 중)

## 배경 / 문제의식

- 처음엔 몬스터는 기본공격 없이 스킬만으로 구현하려 했음
- 그런데 유저가 보기엔 몬스터의 "스킬"도 사실상 기본공격처럼 보임
- 몬스터 공격도 즉시 발동이 아니라 **Spine 애니메이션 구독(키프레임)** 을 통해 진행되어야 함
- 단, 몬스터는 무기를 가진 개념이 아니라서 WSC에 묶기엔 부적절
- 한 몬스터 내에서도 **어떤 공격은 Spine 이벤트 기반, 어떤 공격은 즉시 발동형**이 섞여 있음 (예: 근접 베기는 hit 이벤트 대기, 투사체 소환은 즉시 발동)
- 몬스터는 1~3개의 공격 패턴을 가지며, 이전 레퍼런스처럼 **Rate 기반 랜덤 선택** 구조 예정

## 결정된 방향

### 1. 몬스터 공격 = SSC의 스킬로 처리
- 새 컴포넌트(MAC 등) 만들 필요 없음
- SSC는 그대로 사용, Rate 기반 랜덤 선택만 추가
- 플레이어의 기본공격(WSC)은 기획상 스킬이 아니므로 그대로 유지 (WSC는 플레이어 전용)

### 2. SpineController가 직접 await 헬퍼 제공 (이벤트 구독 방식 폐기)
- SSC/WSC 양쪽 모두 "애니메이션 이벤트 시점까지 대기" 동작이 공통으로 필요함
- 그러나 **이벤트 구독(`+=/-=`) 방식이 아님**. SpineController 내부에 await 가능한 헬퍼를 두고, 호출자는 단순히 await만 함
- BaseSkill·WSC는 SpineController의 이벤트를 구독하지 않음. 구독 수명 관리, 핸들러 함수, 상태 가드 모두 불필요

**SpineController 헬퍼 (개념)**:
```csharp
// 함수명은 가안
UniTask WaitForAnimEvent(string eventKey, CancellationToken ct);
UniTask WaitForComplete(CancellationToken ct);
void Play(string animKey, bool loop);
```

**스킬 사용 예 (Fireball.cs 같은 개별 스킬 클래스)**:
```csharp
// 히트형 스킬 (근접 베기)
spine.Play("Attack_Slash", false);
await spine.WaitForAnimEvent("hit", ct);
DealDamage();
await spine.WaitForComplete(ct);

// 즉시형 스킬 (투사체)
spine.Play("Attack_Fireball", false);
await spine.WaitForComplete(ct);
SpawnProjectile();
```

- 동작 원리: SpineController가 내부적으로 TCS를 보유. 재생 중인 애니의 이벤트 키를 받으면 매칭되는 대기 TCS를 resolve
- WSC도 동일한 방식. WSC가 기본공격 콤보를 돌릴 때 `await spine.WaitForAnimEvent("hit", ct)`로 히트 타이밍 대기, `WaitForAnimEvent("combo_ready", ct)`로 캔슬 시점 대기

**채택 이유 (접근 방식 A의 장점은 그대로 유지)**:
1. 이미 리플렉션으로 개별 스킬 클래스를 만드는 구조라 "스킬마다 코드" 부담은 이미 받아들인 상태
2. 보스 패턴·고유 스킬의 복잡도(다단 히트, 투사체+히트 혼합, 조건 분기)를 템플릿으로 가두면 금방 한계
3. 이벤트 구독 보일러플레이트(`+=/-=`, try/finally, 핸들러 함수, pendingTcs 필드)가 SpineController 내부로 일원화 → 호출부는 선형 코드

### 3. 공통 인터페이스 ISpineController 도입
- 플레이어에는 `PlayerSpineController`가 이미 존재, 몬스터도 자기 SpineController 필요
- 스킬·WSC가 owner의 SpineController에 접근해야 하므로 공통 인터페이스로 추상화

```csharp
interface ISpineController {
    void Play(string animKey, bool loop);
    UniTask WaitForAnimEvent(string eventKey, CancellationToken ct);
    UniTask WaitForComplete(CancellationToken ct);
}
```

- `FieldCharacter`가 `ISpineController`를 노출 → 스킬/WSC가 `owner.SpineController`로 접근
- 이벤트(Action)는 외부에 노출하지 않음. SpineController 내부에서만 Spine raw 이벤트를 받아 TCS 처리

### 4. 애니메이션 재탕으로 스킬화 가능 (A안의 큰 장점)
- **애니메이션 = 단순 리소스 키**. 누가 구동하느냐에 따라 기본공격/스킬로 달라짐
- 같은 `BasicAttack_Swing_1` 애니를 스킬이 재생하면 스킬 데미지/이펙트가 적용됨
- 데미지 계수·판정 범위·이펙트는 각자의 데이터(WeaponSettingData vs SkillSettingData)에서 읽음
- **이점**:
  - 리소스 한계 돌파 — 같은 애니로 등급 다른 스킬 여러 개 파생
  - "N타 기본공격 애니를 그대로 쓰는 피니셔 스킬" 자연스럽게 구현
  - 몬스터는 애니 셋만 맞추면 리소스 1개로 공격 2~3개 파생

### 5. 이벤트 처리: SpineController 내부에 TCS 일원화 (호출부는 await만)
- 처음엔 BaseSkill/WSC가 직접 `+=/-=`로 이벤트 구독하는 안을 검토 → try/finally로 구독 수명 관리해야 해서 오버엔지니어링
- 중간에 "키프레임에 `attack_hit` / `skill_hit` 두 이벤트 분리" 아이디어 나왔으나 **근본 문제 해결 안 됨**:
  - 애니 클립은 자기가 왜 재생되는지 모름 → 두 이벤트 모두 발사 → WSC가 스킬 재탕 애니에서도 기본공격 데미지 쏘게 됨
  - 애니메이터가 공유 애니마다 두 이벤트 빠짐없이 박아야 하는 작성 부담
- **최종 해법**: **이벤트 구독 자체를 SpineController 내부로 캡슐화. 외부는 await 헬퍼만 사용**

```csharp
// SpineController 내부 (개념)
private Dictionary<string, UniTaskCompletionSource> pendingWaits;

// Spine raw 이벤트 수신 (내부)
void OnSpineEvent(string eventKey) {
    if (pendingWaits.TryGetValue(eventKey, out var tcs)) {
        pendingWaits.Remove(eventKey);
        tcs.TrySetResult();
    }
}

public UniTask WaitForAnimEvent(string eventKey, CancellationToken ct) {
    var tcs = new UniTaskCompletionSource();
    pendingWaits[eventKey] = tcs;
    return tcs.Task.AttachExternalCancellation(ct);
}
```

```csharp
// 호출부는 그냥 선형 코드
spine.Play("Attack_Slash", false);
await spine.WaitForAnimEvent("hit", ct);
DealDamage();
```

**왜 깔끔한가**:
- 호출부(SSC/WSC/스킬)는 `+=/-=`/try/finally/핸들러 함수/pendingTcs 필드 전부 없음
- "어떤 이벤트가 발사되면 누구를 깨울지"는 SpineController가 단일 진실. 가드 로직 분산 없음
- 게임 기획상 공격과 스킬은 동시에 일어나지 않으므로 동일 이벤트 키에 대기자가 둘일 일이 없음 → 단순 Dictionary로 충분
- 애니메이터는 `hit` 이벤트 하나만 찍으면 됨

## SpineController의 cancel-on-Play 규칙: 안전망 vs 게임 규칙

- SpineController는 어떤 `Play` 계열 함수가 호출되든 진입 시 모든 pending TCS(`pendingEventWaits`, `pendingCompleteWait`)를 `TrySetCanceled`로 정리한다
- 결과적으로 WSC가 콤보 await 도중에 누가 `PlayDash`/`PlayJump` 등을 호출하면, await가 `OperationCanceledException`으로 throw → WSC의 콤보 루프 finally에서 상태가 자동 정리되며 공격이 취소된다

**주의 — 이건 어디까지나 안전망이지 게임 규칙이 아니다**:
- 사실상 "마지막 Play 호출자 승" 정책. SpineController는 게임 규칙(공격 중엔 dash 금지 등)을 알지 않는다
- 실제 게임 규칙("공격 중엔 dash 입력 무시", "공격 중 슈퍼아머" 등)은 **호출 측(PlayerController)이 `WSC.IsAttacking`을 체크해서 입력 자체를 막는** 형태로 구현해야 한다
- 즉 책임 분리:
  - SpineController: "동시에 두 애니가 재생되지 않게" + "취소된 await는 깨끗하게 던진다" (인프라)
  - PlayerController / 상태머신: "공격 중엔 무엇이 가능하고 불가능한가" (게임 규칙)
- 이 가드는 별도 작업("상태머신과 공격의 통합")에서 추가될 예정. 이번 BSC/WSC 기초 보완 작업의 비범위.

## 남은 논의 사항 (다음 대화에서 이어서)

### Rate 기반 몬스터 스킬 선택 로직 위치
- **옵션 1**: SSC 안에 통합 — SSC에 `UseRandomSkill()` 메서드, Rate 테이블도 SSC가 관리. 플레이어는 키 직접 지정, 몬스터는 랜덤 선택
- **옵션 2**: 몬스터 AI 레벨 — 별도 `MonsterAI`/`MonsterBrain`이 Rate 로직 돌려 `SSC.UseSkill(key)` 호출. SSC는 선택 로직 모름 (관심사 분리상 깔끔)
- **옵션 3**: MonsterCharacter 자체가 보유 — AI 없이 직접 관리
- 현재 프로젝트에 `MonsterAI` 류 개념이 있는지 확인 필요

### 기타 검토 필요
- MonsterSpineController 신규 작성 필요
- 몬스터 스킬의 쿨타임 + Rate 복합 선택 로직
- 2단 히트·복합 타이밍 스킬의 실제 사용 사례 확인
